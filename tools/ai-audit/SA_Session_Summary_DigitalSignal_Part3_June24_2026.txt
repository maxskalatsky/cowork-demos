TASK C — DUAL-MODE SCORING (bf01a43)
─────────────────────────────────────────────────────────────

ARCHITECTURE:

Every submitted URL routes through classifyEntity() before any scoring runs. This is a Claude Haiku call (max_tokens=10) that returns exactly "private_company" or "known_brand". On any API failure or missing key it defaults to "private_company" — the existing pipeline is never broken by a classification error.

classifyEntity() is called after the KV email gate (repeat emails are blocked before the expensive classification call runs) and before the main try block.

Known brand definition passed to Claude:
  publicly traded company, Fortune 1000, major media or sports property,
  government entity, utility, or globally recognized consumer brand.
  Everything else = private_company.

PRIVATE COMPANY PATH: Completely unchanged from Part 2.

ENTERPRISE PATH (known_brand):
1. fetchAgentSignals() — crawl with Scrapfly/Puppeteer fallback
2. dataForSeoOnPage() — wrapped in try/catch, falls back to empty {checks:{},meta:{}} so scoreEnterprise() can run from HTML alone
3. scoreEnterprise() — four dimensions, each tier 1–5

5. getEnterpriseBenchmarkSignal() — Sonnet 4.6, two sentences, benchmark framing for private operator studying the brand

Front end detects r.entityType === 'known_brand' and calls renderEnterprise(r) instead of the private company render path. Section labels update dynamically. No compare box in enterprise mode.

ENTERPRISE DIMENSION SCORING (scorer.js, scoreEnterprise()):

Brand Authority: starts at 8 (Dominant baseline), deducts for failures.
  No JSON-LD:      -2
  Not HTTPS:       -4
  No canonical:    -1
  No description:  -1
  Tier: >=7→1, >=5→2, >=3→3, >=1→4, else→5

Audience Clarity: additive from 0.
  Link count >=30: +3, >=15: +2, >=7: +1
  h2 count >=8: +3, >=4: +2, >=2: +1
  Content-type keyword diversity (news/video/live/fantasy/scores/sports/podcast/etc.) up to +3
  All three heading levels present: +1
  Tier: >=10→1, >=6→2, >=3→3, >=1→4, else→5

AI Visibility: additive from 0.
  llms.txt present: +3
  AI crawlers not blocked: +2
  JSON-LD: +2
  FAQ/Article/HowTo schema: +1
  H1+H2 both present: +1
  Word count >=300: +1
  Tier: >=8→1, >=6→2, >=4→3, >=2→4, else→5

Content Depth: additive from 0.
  /news|blog|stories|articles section path in HTML: +2
  /video|videos|watch section path: +1
  /scores|standings|results section path: +1
  /insights|resources|learn|knowledge|research section path: +1
  Link count >=50: +2, >=20: +1
  h2 count >=6: +2, >=3: +1
  Word count >=1000: +1
  Tier: >=7→1, >=4→2, >=2→3, >=1→4, else→5

PROBLEM 2 — SCORING CALIBRATION:
Three dimensions were scoring incorrectly for ESPN.com:

Brand Authority was T3 because the original rubric added points from zero for signals like sameAs, which ESPN happened to have but most Fortune 500 brands would not reliably trigger. Fixed by inverting to start-from-8 penalize-for-failures approach. A functioning known brand should be Dominant by default.

Audience Clarity was T4 because the original rubric used B2B pattern matching only ("for enterprise|teams|individuals|developers"). ESPN has none of that vocabulary. Fixed by replacing with link count + h2 breadth + content-type keyword diversity that includes consumer media terms (news, video, live, fantasy, scores, standings, highlights, sports, podcast).

Content Depth was T3 because the original rubric measured home-page word count from scraped HTML. ESPN's home page shell is thin. Fixed by replacing with section path detection in HTML links (/news, /video, /scores, etc.) and total link count breadth, which correctly identifies ESPN as having deep content (T1).

TASK F — FORWARD SIGNAL TRUNCATION FIX (f82730e)
─────────────────────────────────────────────────────────────

PROBLEM:
sanitizeForwardSignal() had a 280-character hard truncation at the end. For maherlawpllc.com, the first sentence alone was 332 characters, so the signal was cut mid-word at char 280. This produced broken output.

FIX:
Removed the 280-char hard truncation entirely from sanitizeForwardSignal(). The function now:
1. Strips em dashes, asterisks, backticks, pound signs.
2. Finds the second sentence-terminal punctuation mark ([.!?] followed by whitespace or end-of-string).
3. Returns everything up to and including that punctuation — complete, no mid-sentence cuts.

Both getForwardSignal (private company) and getEnterpriseBenchmarkSignal received the same system prompt update:
  "Write exactly two complete sentences.
   If your output is more than two sentences, return only the first two complete sentences.
   Do not truncate output under any circumstances — every sentence must be complete."

max_tokens raised 250→400 for both calls to ensure two complete sentences are never cut by the token ceiling.

ARCHITECTURAL DECISIONS AND RATIONALE
─────────────────────────────────────────────────────────────

WHY classifyEntity() before the main try block:
The private company path requires DataForSEO and will throw if it fails. The enterprise path wraps DataForSEO in its own try-catch. Calling classifyEntity() before the shared try block lets the two paths fail independently without one masking the other.

WHY default to private_company on classification failure:
The existing pipeline is the known-good path. An API failure should never break an audit for a site that would have worked fine before dual-mode was introduced.

WHY brand authority starts from 8 not 0:
Additive scoring from zero requires the auditor to observe rare signals (sameAs, structured data with specific properties) to reach Dominant. A Fortune 500 brand that simply passes HTTPS and has JSON-LD should read Dominant — that is the observable truth. Deducting for hard failures is more honest than rewarding for rare signals.

WHY content-type diversity for audience clarity:
The previous B2B-only rubric would always score consumer media brands low. ESPN, NYT, ESPN-type properties have massive, clearly-segmented audiences — they just don't use enterprise vocabulary. Link count and h2 breadth measure segmentation directly without being category-biased.

WHY section path detection for content depth:
Home-page HTML word count is a proxy for content depth that breaks on shell-rendered sites and large properties that put everything below the fold. Section paths (/news, /video, /scores) present in href links are a reliable indicator that deep content exists, regardless of what the home page renders.
